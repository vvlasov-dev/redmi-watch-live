# Redmi Watch 5 Active → ПК: данные в реальном времени

Самостоятельный **порт протокола Xiaomi из Gadgetbridge на Windows/Python**. Подключается
к часам по Bluetooth Classic (SPP / COM-порт), проходит шифрованную аутентификацию,
включает realtime-режим и выдаёт **живой пульс и шаги** в консоль, в CSV и в веб-панель.

> Это производная работа от Gadgetbridge (лицензия **AGPLv3**). Используйте для своих
> личных данных и устройства. Благодарность проекту Gadgetbridge — весь протокол реверсили они.

рукопожатие). Чего нельзя проверить без железа: точную последовательность инициализации
конкретной прошивки. Это доводится итеративно на устройстве.

## Два обязательных условия

1. **Auth-ключ (16 байт).** Часы не пускают без ключа, который выдаётся при привязке в
   приложении Mi Fitness. Чисто из-под Windows его получить нельзя — нужен **один раз Android**:
   - Установить Mi Fitness, привязать часы.
   - Достать ключ (методы в вики Gadgetbridge «Pairing types and Auth Key»): обычно через
     `adb logcat` во время привязки, либо из базы/логов приложения, либо привязав часы в
     самом Gadgetbridge и скопировав сохранённый `authkey`.
   - Ключ — 32 hex-символа (можно с `0x` или без).
2. **Канал данных у часов один.** Чтобы читать с ПК, закройте Mi Fitness и/или отвяжите
   часы от телефона (на часах: «подключить новое устройство»). Иначе ПК не получит SPP-канал.

## Установка и запуск (Windows)

```powershell
cd redmi_watch5_live
py -m pip install -r requirements.txt

# 1) Спарить часы в настройках Bluetooth Windows (Classic), затем найти COM-порт:
py client.py --list

# 2) Запуск (подставьте свой COM-порт и auth-ключ):
py client.py --port COM8 --auth-key 00112233445566778899AABBCCDDEEFF --serve 8765 --csv live.csv
```

- Консоль: строки `HR=72 bpm  steps=1234  cal=56`.
- Веб-панель: <http://127.0.0.1:8765> — крупный пульс + шаги, обновление раз в секунду.
- `live.csv` — журнал; его же можно открыть в `../redmi_watch5_dashboard.html` для истории.

Отладка протокола: добавьте `--debug` (печатает hex RX/TX).

## Как это работает

```
client.py   — оркестратор: открывает COM-порт, ведёт стейт-машину, печать/CSV/веб-панель
spp.py      — кадрирование SPP: v1 (BA DC FE … EF) для запроса версии; v2 (A5 A5 + CRC-16/ARC)
xcrypto.py  — рукопожатие miwear-auth: HMAC-SHA256 key-expansion → ключи; AES-CCM (devinfo) и AES-CTR (данные)
miniproto.py— минимальный protobuf-кодек (без зависимостей) для нужных сообщений
selftest.py — оффлайн-проверка всего перечисленного
```

Последовательность подключения:

1. Открыть COM-порт → отправить **V1-запрос версии** (`channel=Version`).
2. Часы отвечают версией; если ≥2 → переключиться на **SPP v2** и отправить session-config.
3. На ответ сессии → **рукопожатие**: phone-nonce → watch-nonce → вывод ключей → AuthStep3.
4. После `AUTHENTICATED` → отправить **enable realtime** (`type=8, subtype=45`).
5. Часы шлют события `type=8, subtype=47` с `RealTimeStats{ steps, heartRate, calories }` → панель.

---

## Если не заработает с первого раза

- **`AUTH FAILED`** → неверный auth-ключ или часы уже перепривязались (ключ сбросился). Переснимите ключ.
- **Нет COM-порта в `--list`** → часы не спарены как Classic, либо нужен «исходящий» COM-порт SPP в настройках Bluetooth.
- **Тишина после сессии** → возможно, прошивке нужен иной порядок инициализации; запустите с `--debug` и пришлите hex — поправим стейт-машину.
- **Часы используют SPP v1 для данных** → этот билд реализует v2; допишем v1-путь при необходимости.
